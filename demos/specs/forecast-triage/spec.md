# Feature: Forecast listing + triage

## Goal
Users can browse a synthetic set of forecasts, inspect a single forecast, create a new forecast entry, and update the forecast's triage status.

## Scope and constraints
- This sprint covers the server API contract only. Frontend changes can consume the API, but UI work is not required for the feature to be considered done.
- Use synthetic data only. Do not call an external weather provider and do not introduce a database dependency.
- Forecasts must be stored in memory for the lifetime of the running process.
- Seed the API with a deterministic synthetic dataset on startup so IDs remain stable for the lifetime of the process.
- A process restart may reset the dataset back to the synthetic seed state.
- Authentication and authorization are out of scope for this sprint.
- Paging, filtering, and sorting controls are out of scope for this sprint.

## API contract

### Forecast shape
Each forecast returned by the API must include:
- `id`: unique integer identifier generated by the server
- `date`: ISO 8601 date string in `yyyy-MM-dd` format
- `temperatureC`: integer value in Celsius
- `temperatureF`: integer value derived from `temperatureC` on the server
- `summary`: non-empty string
- `triageStatus`: one of `new`, `reviewed`, or `escalated`

### `GET /api/weatherforecast`
- Returns `200 OK` with an array of forecasts.
- Returns all forecasts currently held in memory.
- Results must be ordered by `date` ascending, then `id` ascending.

### `GET /api/weatherforecast/{id}`
- Returns `200 OK` with a single forecast when the ID exists.
- Returns `404 Not Found` when the ID does not exist.
- Error responses must use `ProblemDetails`.

### `POST /api/weatherforecast`
- Accepts JSON with `date`, `temperatureC`, and `summary`.
- The client must not supply `id` or `temperatureF`.
- `triageStatus` is optional on create and defaults to `new` when omitted.
- Returns `201 Created` with the created forecast in the response body and a `Location` header pointing to `GET /api/weatherforecast/{id}`.
- The created forecast must be visible from both `GET /api/weatherforecast` and `GET /api/weatherforecast/{id}` immediately after creation.
- Returns `400 Bad Request` with `ProblemDetails` when validation fails.

### `PATCH /api/weatherforecast/{id}/triage`
- Accepts JSON with a required `triageStatus` field.
- Only `new`, `reviewed`, and `escalated` are valid values.
- Returns `200 OK` with the updated forecast.
- Returns `404 Not Found` when the ID does not exist.
- Returns `400 Bad Request` with `ProblemDetails` when the payload is missing or invalid.
- The updated triage status must be visible from both list and detail endpoints immediately after the update.

## Validation rules
- `summary` is required, must not be blank or whitespace, and must be at most 100 characters.
- `temperatureC` must be within `-100` to `100` inclusive.
- `date` is required and must be a valid calendar date.
- Request payloads with unknown or read-only fields required to be server-managed, such as `id` or `temperatureF`, must be rejected with `400 Bad Request`.

## Acceptance criteria
- `GET /api/weatherforecast` returns `200 OK` and the full in-memory forecast list using the documented response shape.
- `GET /api/weatherforecast` returns forecasts in deterministic order by `date` then `id`.
- `GET /api/weatherforecast/{id}` returns `200 OK` for an existing forecast and `404 Not Found` with `ProblemDetails` for an unknown ID.
- `POST /api/weatherforecast` returns `201 Created`, assigns a server-generated ID, defaults `triageStatus` to `new` when omitted, and makes the new item available from subsequent reads in the same process.
- `PATCH /api/weatherforecast/{id}/triage` updates only the triage state and returns the updated forecast.
- Validation failures for create and triage requests return `400 Bad Request` with `ProblemDetails`.
- Read-after-write behavior must be immediate. If response caching is enabled for forecast endpoints, it must be bypassed or invalidated so list and detail responses do not serve stale data after create or triage updates.

## Required tests
- Integration test: `GET /api/weatherforecast` returns `200 OK`, the expected JSON fields, and deterministic ordering.
- Integration test: `GET /api/weatherforecast/{id}` returns the matching forecast for a seeded ID.
- Integration test: `GET /api/weatherforecast/{id}` returns `404` with `ProblemDetails` for an unknown ID.
- Integration test: `POST /api/weatherforecast` with a valid payload returns `201 Created`, includes a `Location` header, and the created forecast is immediately readable from list and detail endpoints.
- Integration test: `POST /api/weatherforecast` rejects invalid payloads such as blank summary, out-of-range temperature, missing date, or client-supplied server-managed fields.
- Integration test: `PATCH /api/weatherforecast/{id}/triage` updates the triage status and the change is immediately visible from list and detail endpoints.
- Integration test: `PATCH /api/weatherforecast/{id}/triage` returns `400` for an invalid status and `404` for an unknown ID.
- Unit test: server-side forecast mapping or model logic correctly derives `temperatureF` from `temperatureC`.
